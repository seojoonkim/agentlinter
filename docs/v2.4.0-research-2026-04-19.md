# AgentLinter v2.4.0 Research (2026-04-19)

> Research brief grounding v2.4.0 in the **2026-04 agent ecosystem** — standards, competitive tools, dogfooding the 5-sibling workspaces, and gaps worth turning into new rule categories.
>
> Scope: research only. No code changes, no repo edits. Dry-run analysis of 5-sibling `AGENTS.md` corpora, web research across 5 axes, final v2.4.0 update proposal.

---

## Executive Summary

1. **AGENTS.md is now the dominant cross-tool standard** (Linux Foundation / Agentic AI Foundation, 60k+ repos, 20+ tools, Codex enforces 32 KiB size cap, Princeton study: -28.6% runtime / -16.6% tokens). AgentLinter must treat AGENTS.md as first-class and validate against the actual spec (size, structure, overlap with README).
2. **SKILL.md + "unified extensibility" convergence** — Claude Code, Codex, Copilot, Cursor all converge on SKILL.md frontmatter (`name`, `description`, `allowed-tools`, `disable-model-invocation`, `user-invocable`). Skills replace commands (2026). AgentLinter's current `skillSafety` + `skillsVsCommands` need modernization.
3. **Harness engineering** (Hashimoto / OpenAI / Chroma): "the system, not the model, is what matters." Key findings — **repo hygiene** and **ADRs** prevent rot; **deterministic tools > LLM for linting**; **agents are lazy** (`--no-verify`, `lint-disable` abuse). This directly validates AgentLinter's market thesis.
4. **Context Rot (Chroma, 18 frontier models)** — all LLMs degrade as context grows. Redundant AGENTS.md duplicating README content **increases** cost by 23%. This is a missing rule category.
5. **Indirect prompt injection (IPI)** is now the #1 agent attack vector in 2026 (Google Calendar/Gemini, Gmail invites, Workspace). HiveFence/prompt-guard patterns are directionally right — but the gap is **data-boundary rules** inside AGENTS.md itself (untrusted external sources labeling, tool manifest lint).
6. **Dogfooding 5 siblings**: 4 out of 5 workspaces are missing explicit "4-element delegation spec" guidance despite it being a TIER-1 rule. All 5 have 14–26 mentions of "sub-agent" but **zero structured orchestration schema**. Clear rule candidate.
7. **Promptfoo was acquired by OpenAI ($86M, 2026-03-09)** — agent red-teaming is now mainstream. AgentLinter's position is complementary (static lint, not runtime fuzzing) but should integrate lightweight red-team hooks.

**Bottom-line recommendation**: the prior v2.4.0 plan's "depth over breadth" thesis is correct. The Multi-Agent Orchestration Linter (#1) and Runtime Signal Integration (#2) remain the right bets, but the research surfaces **5 new concrete rule candidates** and **1 entirely new category** (`context-hygiene`) that should join v2.4.0.

## Axis 1: Agent Framework Landscape (2026-04)

### AGENTS.md — the cross-tool standard

- **Stewardship**: Agentic AI Foundation under the Linux Foundation (morphllm, 2026-03).
- **Adoption**: 60,000+ GitHub repos; 20+ compatible coding agents (Codex, Claude Code, Copilot, Cursor, Windsurf, Devin, Aider, Gemini CLI, Antigravity v1.20.3+ 2026-03-05).
- **Hard constraints**:
  - Codex enforces **32 KiB default size cap** (silent truncation beyond).
  - Nearest-file-wins hierarchy (no @imports unlike CLAUDE.md's 5-level).
  - `AGENTS.override.md` = Codex-specific local override.
- **Evidence-based "what belongs / what doesn't"** (GitHub engineering team, 2,500 repos):
  - ✅ Non-obvious commands with flags, rules that differ from defaults, architecture constraints, explicit boundaries, project-specific gotchas.
  - ❌ Commands already in `package.json`, standard language conventions (Prettier / PEP 8), full API docs, obvious practices ("write clean code"), README duplication.
- **Princeton study (arxiv 2601.20404)**: with AGENTS.md → median runtime -28.6%, tokens -16.6% (n=124 PRs, 10 repos, Codex gpt-5.2-codex).
  - **Follow-up finding**: auto-generated AGENTS.md duplicating README actually **reduced** task success and **increased** cost by 23%.

### SKILL.md — portable skills across ecosystem

- **Unified model (Claude Code 2026)**: `.claude/commands/` and `.claude/skills/` merged; every skill gets a `/slash-command` interface.
- **Frontmatter fields** (cross-tool minimum):
  - `name` (lowercase, hyphens, max 64 chars)
  - `description` (used for auto-invocation ranking)
  - `allowed-tools` (tool whitelist)
  - `disable-model-invocation` (boolean — prevent auto-invoke)
  - `user-invocable` (boolean — visibility in `/` menu)
- **Progressive disclosure**: only frontmatter loaded at session start; body loaded on match; assets loaded on demand.
- **Storage locations**:
  - Project: `.github/skills/`, `.claude/skills/`, `.agents/skills/`
  - Personal: `~/.claude/skills/`, `~/.copilot/skills/`, `~/.agents/skills/`
  - Installable: `npx skills add <url>` (Vercel skills CLI)
- **Tool support matrix**: Codex (Yes), Claude Code (Yes), Copilot (Yes), Cursor (Yes), Gemini CLI / Windsurf (No). SKILL.md is a narrower standard than AGENTS.md.

### Claude Agent SDK — 2026 changes

- **Deferred Tool Loading**: MCP tool schemas no longer loaded at startup — names only, schemas fetched on demand via `ToolSearch`. For 50+ tool environments this is an **order-of-magnitude** context reduction.
- **Streamable HTTP transport** (replaces SSE) + **OAuth 2.1 with PKCE** for remote MCP servers.
- **Hierarchy**: enterprise → user (`~/.claude/CLAUDE.md`) → project (`./CLAUDE.md`); `~/.claude/rules/` splits global instructions across files.
- **`CLAUDE.local.md`** = gitignored personal layer alongside `CLAUDE.md`.
- **Scheduled Tasks** = cloud-based cron triggers for headless runs.

### ACP — Agent Client Protocol

- **Stewards**: JetBrains + Zed (2026). Production spec at `agentclientprotocol.com`.
- **Scope**: standardizes editor ↔ coding-agent communication (local and remote). Parallel to MCP (agent ↔ tool) — ACP sits at a different layer.
- **Key primitives**: session management, turn-based message exchange, tool-call forwarding, streaming deltas.
- **Relevance to AgentLinter**: ACP does not standardize instruction files, so AgentLinter remains complementary; but ACP-compatible agents may need new conformance checks (e.g., "if `acp.config.json` present, validate session boundaries").

### Multi-agent frameworks — 2026 snapshot

| Framework | Orchestration style | Observable manifest |
|---|---|---|
| **LangGraph** | Graph-based control flow, explicit state, strong observability | `langgraph.json` + graph definitions |
| **CrewAI** | Role-based teams, sequential/hierarchical | `crew.yaml` + role/task specs |
| **AutoGen (Microsoft)** | Conversational, peer-agents | Python-centric, no YAML manifest |
| **OpenAI Agents SDK** | Assistants + handoffs, guardrails built-in | `assistant.json` + tool configs |

**Common emerging pattern**: role/task/handoff specs + guardrails + memory. No single standardized "orchestration manifest" yet, but **role+goal+constraints+completion** schema is de facto universal. This is the shape AgentLinter's `orchestration` category should encode.

**Takeaway for AgentLinter v2.4.0**:
- Enforce 32 KiB AGENTS.md cap (warning at 24 KiB, error at 32 KiB).
- Detect README duplication (fuzzy-match against sibling `README.md`).
- Validate SKILL.md frontmatter schema (required: `name`, `description`; recommended: `allowed-tools`).
- Detect MCP "deferred loading" readiness (warn when AGENTS.md lists 50+ MCP tools as always-loaded).
- Add `CLAUDE.local.md` gitignore check.

## Axis 2: Quality Standards & Industry Signals

### Stanford AI Index 2026 (9th edition, 400+ pages)

Key takeaways relevant to agent lint:
- Agent capability is accelerating and reaching more users than ever; model performance continues to improve across benchmarks.
- Responsible-AI chapter highlights a growing gap between **capability** and **safety evals** — most production agents ship without systematic adversarial testing.
- Economy chapter flags **agent deployment cost** as a rising line item — token spend per task is the dominant factor. This validates AgentLinter's `token-budget` and `tokenEfficiency` rules and argues for elevating **cost-aware linting** in v2.4.0.

### Harness Engineering best practices (Sakasegawa / Hashimoto / OpenAI / Chroma, 2026-03)

This stream of work is the most directly relevant to AgentLinter's thesis:

- **Hashimoto's definition**: harness engineering = continuous improvement of AGENTS.md + toolchain that lets an agent self-verify its own work. AgentLinter *is* the toolchain instantiation of this.
- **Investing in a harness compounds**: one linter rule prevents a class of mistakes across every future session. This is the explicit argument for a linter over one-off instructions.
- **Repo hygiene — design for rot**:
  - Agents treat every piece of text as equally authoritative. Stale prose in-repo becomes indistinguishable from the current truth.
  - Belongs in repo: executable artifacts (code, tests, lint configs, type defs, CI), **ADRs** (Architecture Decision Records with status: Accepted/Superseded/Deprecated).
  - Does **not** belong in repo: hand-written architectural prose, long API descriptions, "how the system currently works" docs (they rot).
- **Tests resist rot better than documentation** — every agent mistake should become a test, not a prose warning.
- **Deterministic tools > LLM-as-linter** (HumanLayer / OpenAI): "whenever a deterministic tool can do the job, use it." Offload quality enforcement from CLAUDE.md prompts to mechanical enforcement (lint, format, type check).
- **Agents are lazy** (Tealarson, 2026-03-27): observed behavior of agents abusing `--no-verify` on commits and inserting `lint-disable` comments instead of fixing problems. AgentLinter can surface these exact anti-patterns.

### Teallarson's "linting config > agent instructions" thesis

Direct industry echo of AgentLinter's positioning. Key lines applicable as rule candidates:
- **Do NOT put** tabs-vs-spaces, brackets, camelCase-vs-PascalCase in AGENTS.md. Put them in a linter config. Agents (and humans) should not reason about them.
- Agents abuse `git commit --no-verify`. Rule: detect if AGENTS.md forbids `--no-verify` and pre-commit hook is configured.
- Agents add `lint-disable` comments to dodge errors. Rule: detect if AGENTS.md explicitly forbids lint-disable comments.
- Prefer **agent hooks** (Claude Code, Cursor, Codex all support) for lint/format after `Edit`/`Write` — this is a concrete v2.4.0 rule (`hooks/lint-on-edit`).

### Anthropic Responsible Scaling Policy implications

- Deployment tier escalation requires demonstrable **pre-deployment evals** + **access controls** + **response plan**. AgentLinter can encode a lightweight "RSP-readiness" checklist (e.g., `responsible-ai/incident-response-defined`, `responsible-ai/tool-access-controls-documented`).

**Takeaway for AgentLinter v2.4.0**:
- Add category `context-hygiene` — detect README duplication, stale dates, un-superseded ADRs, prose describing architecture that should be in code.
- Add rule `quality/deterministic-tooling-first` — flag AGENTS.md sections that instruct the agent to do what a linter/formatter should do.
- Add rule `quality/no-verify-ban` — detect absence of an explicit `--no-verify` / `lint-disable` prohibition in AGENTS.md for repos that have pre-commit hooks.
- Add rule `quality/prefer-agent-hooks` — if lint/format instructions are in AGENTS.md but no hook config exists, recommend migrating to hooks.

## Axis 3: Competitive Landscape

### Promptfoo (OpenAI, acquired 2026-03-09 for ~$86M)

- **Scope**: red-teaming / adversarial testing (runtime), 50+ vulnerability types aligned to OWASP LLM Top 10.
- **Architecture**: plugins (what to attack) × strategies (how to deliver: base64, leetspeak, Crescendo multi-turn, GOAT, Hydra adaptive, emoji encoding, "Mischievous User") × LLM-as-judge grading (GPT-5 default).
- **Coverage**: prompt injection (direct + indirect), PII/COPPA/FERPA, RBAC/BOLA, SQL/shell/SSRF injection, excessive agency, bias, harmful content.
- **Adoption**: 18k+ GitHub stars, 350k+ users, 25%+ Fortune 500.
- **Gap vs AgentLinter**: Promptfoo is **runtime** and **behavioral**. AgentLinter is **static** and **structural**. Complementary, not competing.
- **Integration opportunity**: `agentlinter redteam --quick` — spawn a Promptfoo scan with a minimal policy file derived from AgentLinter's `security/*` rule hits.

### Helicone (AI Gateway + observability)

- **Scope**: unified OpenAI-compatible gateway for 100+ LLM providers, self-hostable, integrates with Promptfoo as a provider. Observability + caching + cost tracking, not a linter.
- **Gap vs AgentLinter**: runtime observability vs static structure. Potential input source for Runtime Signal Integration (feature #3 in existing plan) — Helicone logs can feed rule-hotness detection.

### MintMCP / PolicyLayer / Qualys TotalAI — MCP security tooling

- **MCP is the new shadow IT of 2026** (Qualys). Most orgs lack visibility into what MCP servers expose.
- **Rate limiting** is the dominant primitive (per-tool, global, YAML policy files).
- **Not linters**, but surface a category AgentLinter doesn't cover: **MCP tool manifest lint**.
- **Gap**: no tool validates MCP tool declarations in AGENTS.md / settings files for missing rate limits, dangerous operations without confirmation, or tool descriptions that would trigger indirect prompt injection.

### Agentfile, PromptLayer, Anthropic internal tools

- **Agentfile** (Docker-for-agents concept): packaging spec. Too early to validate against, but monitor.
- **PromptLayer**: prompt versioning. AgentLinter could export `CHANGELOG.md` diffs in a compatible format.
- **Anthropic internal `clai-rules`**: not publicly specified; AgentLinter is the public equivalent.

**AgentLinter's unique position (confirmed by gap analysis)**:
- Only static-analysis tool for AGENTS.md / CLAUDE.md / SKILL.md.
- Only tool with rule taxonomy spanning structure + clarity + security + token-budget + runtime + orchestration.
- Only tool targeting the **writer of the file** (vs Promptfoo targeting the runtime of the agent).

**Takeaway for AgentLinter v2.4.0**:
- Add category `mcp-manifest` — validate MCP tool lists: rate-limit annotations, dangerous-op flags, description safety (no imperative text that could be mis-interpreted by a naive agent).
- Add `redteam-handoff` command — emit Promptfoo-compatible policy stub from AgentLinter findings.
- Publish an "AgentLinter vs Promptfoo" positioning doc so they're seen as complementary (static × runtime).

- **Homogeneous anti-patterns across siblings** suggest a **template/bootstrap** mechanism would help. AgentLinter could ship a `agentlinter init --profile=multi-agent` generator.

- New rules 1–8 above are all concrete and land in the proposed `orchestration` + `context-hygiene` categories.
- Immediate non-rule insight for Zeon: at 27.7 KiB, AGENTS.md is one heading cluster away from Codex silent truncation. This alone is worth surfacing as a TIER-1 AgentLinter finding.

## Axis 5: Security / Performance / Cost

### Indirect prompt injection (IPI) — 2026 dominant attack vector

- **Google Gemini + Google Calendar** (Jan 2026, Hacker News disclosure): malicious instructions embedded in calendar event titles bypassed Gemini authorization guardrails, exfiltrated private data. Repeat of the Workspace / Gmail invite pattern (mid-2025).
- **Google's layered defense strategy** (Workspace Security, 2026-04): model hardening + purpose-built ML classifiers + system-level safeguards (per-prompt lifecycle stage).
- **MIT Technology Review (2026-01-28)**: "Rules fail at the prompt, succeed at the boundary." Guardrails must live at system boundaries (tool I/O, data sources), not in prompt text alone.
- **Pattern families to detect in AGENTS.md**:
  - Untrusted external content not wrapped in a `<EXTERNAL_UNTRUSTED_CONTENT>` style boundary.
  - Tool manifests that treat all tool outputs as trusted user instructions.
  - Memory write paths that ingest untrusted sources without sanitization.
  - `skills/` or `memory/` patterns where data sources aren't classified as trusted vs untrusted.

### Context rot (Chroma research, 18 frontier models)

- **All** 18 tested frontier models degrade as context grows. Degradation is non-uniform — hallucinations, position bias, distractor sensitivity.
- **Failure modes** amplify with: conversation history, failed attempts, debug output, noise.
- **Practical impact**: redundant AGENTS.md costs 23% more. Long prose sections in AGENTS.md that are always loaded are pure context tax.
- **Karpathy's "wiki memory" pattern** (referenced in Zeon's `memory/` layer): separate always-loaded (inbox, open-loops) from on-demand (entities, tasks, decisions). AgentLinter v2.4.0 should validate this split where it can detect it.

### MCP tool manifest lint

- **MintMCP / PolicyLayer**: per-tool + global rate limits, YAML policy format.
- **Qualys TotalAI**: MCP = shadow IT. Attack surface discovery is the first defense.
- **Rule candidates for AGENTS.md / `mcporter.json`**:
  - `mcp-manifest/rate-limit-declared` — tools without explicit rate-limit annotation.
  - `mcp-manifest/dangerous-op-flag` — tools invoking external writes (email, payment, code execution) without a confirmation gate.
  - `mcp-manifest/description-safety` — tool descriptions containing imperative text that a naive agent might execute ("always call this before X" pattern = known IPI vector).
  - `mcp-manifest/tool-count-deferred-loading` — if >50 MCP tools configured without deferred loading enabled, warn.

### Cost-aware linting

- Stanford AI Index 2026 + Chroma context-rot both push cost to center.
- AgentLinter's current `tokenBudget` + `tokenEfficiency` rules are the right foundation. v2.4.0 additions:
  - `cost/model-choice-per-task` — flag AGENTS.md sections that pin expensive models (opus) to cheap tasks (status checks, simple reads).
  - `cost/always-on-bloat` — compute a "context cost budget" = AGENTS.md bytes × loading frequency. Emit USD estimate at top of report.
  - `cost/missing-compaction-rule` — no `/compact` or memory-flush policy in AGENTS.md.

### Recursive / self-modifying agent anti-patterns

- Garbage-collection agents (OpenAI harness talk) are recursively susceptible to context rot; Sakasegawa argues they must use deterministic rules only, not agent judgement.
- **Rule candidates**:
  - `security/recursive-agent-loops` — AGENTS.md instructions that could cause an agent to spawn itself unboundedly (missing depth / timeout caps).
  - `security/self-modify-guardrails` — AGENTS.md allowing agent to modify its own AGENTS.md / SOUL.md without human review.

**Takeaway for AgentLinter v2.4.0**:
- Category `security+` expansion is right, but rename to `security/boundary` or split into `security/prompt-injection`, `security/data-boundary`, `security/recursive-safety` — more actionable than a flat HiveFence merge.
- Category `context-hygiene` (new) is the highest-ROI addition, backed by three independent research threads (Princeton, Chroma, OpenAI).
- Category `mcp-manifest` (new) fills a real market gap.
- Enhance `tokenBudget` to a `cost` rule set with USD-denominated estimates.

## 📌 v2.4.0 Final Proposal (research-grounded)

### Updated positioning

> **AgentLinter v2.4.0 — "Harness-Grade Static Analysis for the AGENTS.md era."**
>
> Depth over breadth. Three pillars: **Orchestration**, **Context Hygiene**, **Security Boundaries** — plus a measured Runtime Signal layer as opt-in.

### Categories map

| Status | Category | Focus | Rules (target) |
|---|---|---|---|
| ✅ Existing | structure, clarity, consistency, completeness, memory, runtime, security, blueprint, token-budget, freshness, skillSafety, remoteReady | Baseline static analysis | 60+ |
| 🆕 New | **orchestration** | Multi-agent spec enforcement | 8–10 |
| 🆕 New | **context-hygiene** | Rot prevention, duplication, ADRs | 5–7 |
| 🆕 New | **mcp-manifest** | Tool-manifest safety | 4–5 |
| 🔄 Renamed / split | `security` → **security/prompt-injection** + **security/data-boundary** + **security/recursive-safety** | IPI + boundaries | 30–50 |
| 🔄 Upgraded | `tokenBudget` → **cost** (USD-denominated) | Cost-aware linting | 5–6 |
| ⏸️ Held | HiveFence merge | On hold per 형 direction | — |
| 🟡 Opt-in | **runtime** | Execution-log integration | 3–5 (opt-in) |

## ⚠️ Risks / Open Questions

1. **Heuristic quality for `orchestration/*` rules** — these are prose-pattern-based (keyword + structure), higher false-positive risk than lint rules on code. Mitigation: ship as `info` / `warning`, never `error` initially; tune thresholds after 2-week dogfooding.
2. **`context-hygiene/cross-workspace-duplication` scope** — does AgentLinter have the concept of "sibling workspaces"? Need a new CLI flag (`--siblings <glob>`) or a monorepo mode.
3. **`security/recursive-safety/self-modify-gate`** — if triggered, this is the highest-severity finding AgentLinter can emit. Needs careful prose so users don't panic.
4. **HiveFence plugin boundary** — if prompt-guard becomes a plugin, versioning strategy (semver vs pattern-DB updates) needs definition.
5. **Promptfoo positioning** — after the OpenAI acquisition, Promptfoo may expand into static analysis. AgentLinter should publish its positioning doc early to anchor the static × runtime split in the market narrative.
6. **Size cap rule accuracy** — Codex's 32 KiB cap is documented but may change. Rule should pull cap from a config-driven constant, not hard-code.
7. **Rule count vs depth** — v2.4.0 adds ~25 rules. Current count reaches ~85 rules. At some point users will want preset profiles ("strict" / "balanced" / "minimal"). Consider shipping profiles in v2.4.0 itself.
